Project Architecture: Identity Verification-as-a-Service (VaaS)System Overview
The platform is a decoupled microservice API designed to verify professional credentials (initially doctors). Client platforms purchase API keys to access this verification power. The user experience utilizes a redirect flow: the user is redirected to the verification domain, completes the process, and the system sends a PUT request callback to the client's backend to update the user's status.  Phase 1: Document Ingestion & Verification
When the doctor is redirected to the platform, they upload their documents (diplomas, job letters, certificates).  Documents are converted from PDF to high-DPI images and classified by page type.  Pipeline A (Text Extraction): An OCR model extracts raw text, and an LLM structures it into a JSON containing names, universities, and previous jobs.  Pipeline B (Fraud Detection): The system compares the document layout against legitimate templates. It also crops the stamp region, extracts the dean's name via OCR, and verifies if the dean is real and in role.  Web Scraper Agent: Simultaneously, an agent scrapes specific public registries (like the Algerian Ministry of Health) and platforms like LinkedIn to cross-reference university staff records and hospital data.  Phase 2: The Gatekeeper QuizBased on the extracted text from Pipeline A, the system generates three survey questions.  The doctor must answer these written questions.  If the candidate fails this initial quiz, the process is terminated immediately to save compute resources.  Phase 3: RAG Pipeline PreparationAll data (Pipeline A JSON, Pipeline B verdicts, Scraper reports, and Survey responses) is merged into one unified document per candidate.  This document is split by section (chunking), passed through an embedding model, and stored in ChromaDB.  ChromaDB will act as the Retrieval-Augmented Generation (RAG) vector database for the interview phase.  Phase 4: AI Video Interview & Anti-CheatThe Interview Loop: The agent fetches context from ChromaDB to generate highly specific questions.  Low Latency Audio: The system uses lightweight models—Whisper for Speech-to-Text (STT) and Kokoro for Text-to-Speech (TTS)—to ensure a natural, lag-free conversation.  Anti-Cheat (Eye Tracking): To defeat AI eye-contact spoofing (like Nvidia Broadcast), the React frontend will display a random green dot that the user must look at while answering.  Expression Stream: A live camera feed processes the doctor's expressions, logging emotion labels and timestamps to detect stress or inconsistencies.  Phase 5: Decision & CallbackAn Answer Evaluator compares the spoken text against the ChromaDB "truth" to create a Q&A log.  The Approval Model calculates a weighted score based on document validity, Q&A accuracy, and expression consistency.  Outcomes: The candidate is either Approved, Rejected, or Flagged (if they fail multiple times, they are flagged for human review).  If rejected for fraud, their Face ID is banned (covered legally via a user EULA).  A PUT request is sent to the client's ID to finalize the status.
